# المرجع المعماري لمنصة الحي (System Architecture)

مرحباً بك في الوثيقة الهندسية لتوثيق معمارية وبنية مشروع "الحي" المتكامل لخدمات الـ B2B و הـ B2C.

## 1. فلسفة المشروع (Monorepo)
يعتمد المشروع في بنائه على بيئة `Monorepo` تتيح لنا تجميع 7 تطبيقات و 12 باقة (Packages) تحت مظلة شجرة مشاريع (Git Repository) واحدة.

- **لماذا Monorepo؟**
يسهل تبادل الأكواد بشكل فوري. لو أردنا تغيير لون زر الشراء، فإن التعديل في حزمة ألوان `alhai_design_system` سيتطبق تلقائياً على واجهات الكاشير والمدير والعميل.
- **أداة الإدارة الدورية:** `Melos`. نستخدم أمر `melos bootstrap` لربط الاعتماديات، و `melos run test` لإجراء الاختبارات على كل المستودعات.

## 2. هيكلة الحزم (Packages Map)
يتم ضخ البيانات للتطبيقات عبر الـ Packages التالية:
1. `alhai_core`: تحتوي على الـ Models والـ Data Repositories.
2. `alhai_services`: تحتوي على كل لوجيك العمليات.
3. `alhai_database`: تحتوي على أكواد قواعد Local SQLite (Drift).
4. `alhai_auth`, `alhai_zatca`, `alhai_pos` لترجمة الأجهزة وتشفير الفواتير.

## 3. التخزين والخوادم (Supabase Backend)
جميع بيانات ومتاجر المشروع متصلة بخادم سحابي عالي المتانة (PostgreSQL Server) مدار بواسطة Supabase:
- **تحصين الصلاحيات (Row Level Security - RLS):** تم كتابة قواعد تمنع أن يصل متجر لبيانات أو فواتير عميل بمتجر آخر بشكل قطعي.
- **دوال الحافة (Edge Functions):** العمليات الجبارة كحساب أرباح المنصة (2%) تتم داخلياً في دوال الخادم لتسريع المعالجة.

## 4. نظام الذكاء الاصطناعي (AI Reorder Engine)
تم فصل الذكاء الاصطناعي ليعمل كـ `Micro-service` مستقلة بلغة Python، يقوم بقراءة أنماط البيع التاريخية للمتجر لمعرفة موعد نفاد البضاعة، ويعيد تحفيز لوحات `admin_lite` بتوصيات أوتوماتيكية لأوامر الشراء.
