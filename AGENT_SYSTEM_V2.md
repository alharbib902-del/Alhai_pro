## 1️⃣ Source of Truth (مصدر الحقيقة الوحيد)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   📊 PROD.json = المصدر الرسمي الوحيد لحالة المهام                          │
│                                                                             │
│   ├── completed: true/false    ← هل اكتملت؟                                │
│   ├── blocked: true/false      ← هل متوقفة؟                                │
│   ├── blocker_reason: "..."    ← سبب التوقف                                │
│   └── actual_hours: X          ← الوقت الفعلي                              │
│                                                                             │
│   📝 AGENT_LOG.md = سجل تفصيلي للتوثيق والقراءة البشرية                     │
│   📈 progress.txt = ملخص مرئي (يُولّد من PROD.json)                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**القاعدة:** عند التعارض، PROD.json هو الصحيح دائماً.

## 2️⃣ Definition of Done (DoD) - تعريف الاكتمال

### ✅ المهمة تُعتبر مكتملة فقط إذا:

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   ☐ 1. الكود يعمل (flutter run بدون crash)                                 │
│   ☐ 2. flutter analyze = 0 errors (warnings مقبولة)                        │
│   ☐ 3. flutter test = كل الاختبارات تمر (إذا وُجدت)                        │
│   ☐ 4. PROD.json محدّث: completed=true + actual_hours                      │
│   ☐ 5. AGENT_LOG.md محدّث بالتفاصيل                                        │
│   ☐ 6. (إذا Drift/Freezed) build_runner تم تشغيله                          │
│                                                                             │
│   ❌ بدون هذه الخطوات = المهمة غير مكتملة                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 📋 Checklist التشغيل الموحد:

```bash
# 1. تثبيت التبعيات
flutter pub get

# 2. توليد الكود (إذا استخدمت Drift/Freezed/Injectable)
dart run build_runner build --delete-conflicting-outputs

# 3. فحص الأخطاء
flutter analyze

# 4. تشغيل الاختبارات
flutter test

# 5. تشغيل التطبيق (اختياري للتأكد)
flutter run -d windows  # أو macos أو chrome
```

## 3️⃣ Risk Levels (مستويات الخطورة)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   🟢 low: UI, Polish, simple features                                       │
│      → تنفيذ عادي                                                           │
│                                                                             │
│   🟡 medium: Business logic, Reports, Permissions                           │
│      → اختبار إضافي مطلوب                                                   │
│                                                                             │
│   🔴 high: Database, Sync, Payments, ZATCA                                  │
│      → مراجعة مطلوبة + اختبار مكثف + توثيق تفصيلي                           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### حقول PROD.json الجديدة:

```json
{
  "id": "SYNC-001",
  "risk_level": "high",      // low | medium | high
  "review_required": true,   // المهام الحرجة تحتاج مراجعة
  "blocked": false,
  "blocker_reason": null
}
```

### المهام عالية الخطورة (High Risk):
- `SETUP-002` - Drift Database
- `SETUP-003` - Sync Queue
- `SYNC-001`, `SYNC-002` - Data Sync
- `POS-002`, `POS-003` - Payments & Sales
- `ZATCA-001` - Tax Integration
- `PAY-001` - Payment Devices
- `RET-001` - Returns/Refunds

### قواعد المهام عالية الخطورة:
1. ✅ اختبار مكثف (unit + integration)
2. ✅ توثيق تفصيلي لكل قرار
3. ✅ مراجعة الكود قبل الانتقال
4. ✅ backup قبل التعديل على DB

## 4️⃣ Dependencies قبل Priority

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   🚫 القاعدة الصارمة:                                                       │
│                                                                             │
│   لا تبدأ أي مهمة حتى تكتمل كل الـ dependencies                            │
│   حتى لو كانت المهمة P0 والـ dependency هي P1                              │
│                                                                             │
│   مثال:                                                                     │
│   ├── UI-002 (P0) depends on SETUP-002 (P0)                                │
│   ├── SETUP-002 غير مكتمل                                                  │
│   └── ❌ لا تبدأ UI-002 حتى يكتمل SETUP-002                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### ترتيب اختيار المهمة:

```
1. فلتر: المهام غير المكتملة (completed=false)
2. فلتر: المهام غير المتوقفة (blocked=false)
3. فلتر: المهام التي اكتملت dependencies
4. رتّب حسب: Priority (P0 → P1 → P2 → P3)
5. اختر: الأولى في القائمة
```

## 4️⃣ نظام Blockers الرسمي

### متى تُعلّم المهمة blocked؟

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   المهمة تصبح blocked إذا:                                                  │
│                                                                             │
│   1. فشلت 3 محاولات متتالية                                                │
│   2. تحتاج قرار من المستخدم/الفريق                                         │
│   3. تنتظر dependency خارجية (API, package, etc.)                          │
│   4. خطأ تقني لا يمكن حله حالياً                                           │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### كيفية التعامل مع Blocker:

```
1. في PROD.json:
   {
     "id": "SETUP-002",
     "completed": false,
     "blocked": true,
     "blocker_reason": "Drift version conflict with Flutter 3.x",
     "blocker_attempts": 3,
     "blocker_date": "2026-02-01"
   }

2. في AGENT_LOG.md:
   سجّل كل المحاولات والأخطاء بالتفصيل

3. انتقل لأقرب مهمة:
   - لا تعتمد على المهمة المتوقفة
   - نفس الأولوية أو أقل
```

## 5️⃣ سياسة Git

### Branch Strategy:

```bash
# فرع لكل Phase
git checkout -b phase-0-foundation

# أو فرع لكل مهمة كبيرة
git checkout -b task/SETUP-002-drift-database
```

### Commit Message Format:

```bash
# الصيغة
git commit -m "TASK-ID: وصف قصير"

# أمثلة
git commit -m "SETUP-001: Project setup with Riverpod and GoRouter"
git commit -m "UI-001: POS screen split layout"
git commit -m "FIX: Drift migration error in products table"
```

### قواعد Git:

```
✅ مسموح:
- commit بعد كل مهمة مكتملة (DoD محقق)
- push للفرع الخاص

❌ ممنوع:
- merge بدون passing tests
- push مباشر لـ main
- commit كود لا يمر flutter analyze
```

## 6️⃣ قواعد Offline-First (خاص بـ POS)

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   🔌 مبادئ Offline-First Design:                                           │
│                                                                             │
│   1. Local First: كل عملية تُكتب في Drift أولاً                            │
│   2. Queue Sync: أي تغيير يُضاف لـ sync_queue                              │
│   3. Idempotent: عمليات الـ sync يمكن تكرارها بأمان                        │
│   4. UI Independence: الواجهة لا تنتظر الشبكة لتكمل                        │
│   5. Conflict Resolution: قواعد واضحة لحل التعارضات                        │
│                                                                             │
│   تدفق البيانات:                                                            │
│   User Action → Drift (Local) → Sync Queue → Supabase (Remote)             │
│                     ↓                              ↓                        │
│                 UI Updates                   Background Sync                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

# ✅ ملخص التعديلات
# ═══════════════════════════════════════════════════════════════════════════════

| التعديل | الأثر |
|---------|-------|
| DoD ثابت | لا خلاف على "مكتمل" |
| PROD.json = Source of Truth | لا تعارض بين الملفات |
