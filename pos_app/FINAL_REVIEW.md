## 1. توثيق المشروع وأهدافه

### 1.1 فكرة المشروع

**تطبيق الحي POS** هو نظام نقاط بيع متكامل مبني بتقنية Flutter، يستهدف السوق العربي (السعودية بشكل أساسي) ويدعم 7 لغات. التطبيق يعمل على منصات متعددة (Android, iOS, Web, Desktop) مع دعم كامل للعمل دون اتصال بالإنترنت (Offline-First).

### 5.1 المشاكل الحرجة

#### 5.1.1 رمز PIN افتراضي `1234` لموافقة المدير

**الملف:** `lib/services/manager_approval_service.dart:44`
**الخطورة:** حرجة

**المشكلة:** خدمة موافقة المدير تقبل `expectedPin = '1234'` كقيمة افتراضية. أي شخص يعرف هذا الرقم يستطيع تنفيذ عمليات حساسة مثل:
- إلغاء الفواتير والاسترجاع
- تعديل الأسعار
- تصدير البيانات

```dart
// الكود الحالي (خطير)
String expectedPin = '1234', // PIN افتراضي للاختبار
```

**الحل:** إزالة القيمة الافتراضية نهائياً وإجبار التحقق من الخادم:

```dart
// الحل
static Future<bool> requestApprovalWithLocalVerification({
  required BuildContext context,
  required String action,
  required String expectedPin, // بدون قيمة افتراضية
}) async { ... }
```

أو حذف الدالة كلياً واستخدام `onVerify` callback مع الخادم في جميع المسارات. يمكن تقييد الدالة بـ `kDebugMode` فقط.

---

#### 5.1.2 تشفير OTP بدون Salt

**الملف:** `lib/services/whatsapp_otp_service.dart:401-405`
**الخطورة:** حرجة

**المشكلة:** رمز OTP المكوّن من 6 أرقام (مليون احتمال فقط) يُشفّر بـ SHA-256 بدون salt. إذا تم الوصول للـ hash المخزّن (عبر نسخة احتياطية للجهاز أو تفريغ الذاكرة)، يمكن كسره في أجزاء من الثانية بتجربة جميع الاحتمالات.

```dart
// الكود الحالي (خطير)
static String _hashOtp(String otp) {
  final bytes = utf8.encode(otp);
  final digest = sha256.convert(bytes);
  return digest.toString();
}
```

**الحل:**
- **الخيار الأفضل:** نقل التحقق من OTP بالكامل إلى الخادم — الجهاز لا يخزّن أي hash
- **الخيار البديل:** إضافة salt عشوائي لكل جلسة:

```dart
static String _hashOtp(String otp, String sessionSalt) {
  final bytes = utf8.encode('$sessionSalt:$otp');
  final digest = sha256.convert(bytes);
  return digest.toString();
}
```

### 5.2 المشاكل عالية الخطورة

#### 5.2.1 التحقق من OTP على جهاز العميل فقط

**الملفات:** `whatsapp_otp_service.dart`, `otp_service.dart`

**المشكلة:** دورة حياة OTP كاملة تحدث على الجهاز:
1. الجهاز يُولّد OTP
2. الجهاز يُرسله عبر WhatsApp API
3. الجهاز يُخزّن الـ hash محلياً
4. الجهاز يتحقق من إدخال المستخدم

مهاجم يملك وصولاً للجهاز يستطيع: قراءة الـ hash وكسره، مسح بيانات التطبيق لإعادة تعيين محاولات الحد، أو اعتراض طلب API لاستخراج OTP بنص صريح.

**الحل:** تنفيذ التحقق من OTP على الخادم. الجهاز يُرسل فقط OTP المُدخَل إلى الخادم الذي يتحقق منه.

---

#### 5.2.2 تشفير XOR بدلاً من AES-256-GCM

**الملف:** `lib/core/security/web_encryption_service.dart:322-337`

**المشكلة:** خدمة التشفير على الويب تدّعي استخدام AES-256-GCM لكنها تستخدم فعلياً XOR مع keystream مشتق من HMAC. هذا أضعف بكثير من AES-GCM الحقيقي.

**الحل:** استخدام حزمة `webcrypto` أو `dart:js_interop` للوصول إلى Web Crypto API الأصلي في المتصفح.

---

#### 5.2.3 مقارنة hash الـ PIN غير آمنة ضد هجمات التوقيت

**الملف:** `lib/core/security/pin_service.dart:156`

**المشكلة:** استخدام المقارنة العادية `==` التي تتوقف عند أول حرف مختلف، مما يسمح لمهاجم بتحديد الـ hash الصحيح تدريجياً عبر قياس أوقات الاستجابة.

**الحل:** استخدام مقارنة ثابتة الوقت (constant-time comparison) كما هو مُطبّق بالفعل في `csrf_protection.dart`.

### 6.2 عمليات حسابية ثقيلة داخل `build()`

تُنفَّذ عمليات فرز `O(n log n)` وتصفية متعددة `O(n)` في كل إعادة بناء للواجهة. هذا يعني أن كل حركة ماوس أو كتابة حرف تُعيد حساب القائمة كاملة.

| # | الشاشة | العمليات | الخطورة |
|---|--------|----------|---------|
| 1 | العملاء | `where()` ×2 → `sort()` → `fold()` → `where().length` | عالية |

## الخلاصة

تطبيق الحي POS يمتلك **بنية تحتية قوية ومدروسة** مع أدوات ومكوّنات عالية الجودة. المشكلة الأساسية ليست في غياب الأدوات بل في **عدم الاتساق في تطبيقها** عبر جميع الشاشات.

المراحل الثلاث الأولى (إصلاحات فورية + حالات الخطأ + RTL) كافية لجعل التطبيق **قابلاً للإطلاق** بثقة. المرحلتان الرابعة والخامسة تُحسّنان الجودة الشاملة وتُعدّان التطبيق **للتوسع**.

**التقييم النهائي: 5.6 / 10** — مع إمكانية الوصول إلى **8+ / 10** بعد تنفيذ المراحل الخمس.
